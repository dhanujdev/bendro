# Health & Safety Rules — Bendro

> Non-negotiable health and safety rules for an exercise / stretching product.
> Bendro guides physical movement. Injury risk is real. These rules are enforced
> at the infrastructure level, not by convention.
> See docs/HEALTH_COMPLIANCE.md (when authored) for the full framework.

---

## Absolute Prohibitions (Cannot Be Overridden)

These behaviors MUST NEVER ship, regardless of:
- Workspace owner / user policy settings
- Special permissions or API flags
- A/B-test or experiment cohorts
- Internal admin overrides

1. **No diagnosing user injuries.** The product NEVER tells a user what is wrong with them medically. It can only suggest "stop and consult a healthcare provider."
2. **No content encouraging self-harm.** Never recommend a stretch through unmanageable pain, never encourage ignoring pain signals, never gamify pushing past pain.
3. **No medical/PT credentials claim.** UI copy and AI-generated text MUST NOT imply Bendro is a licensed physical therapist, doctor, or chiropractor.
4. **No prescription-style instructions.** Never use language like "take this for your back pain" — always frame as general flexibility/mobility content.
5. **No content targeted at minors as therapy.** Bendro is for adult flexibility and mobility. No marketing or content that positions it as a treatment for pediatric musculoskeletal conditions.

## Mandatory Disclosures (Always Attached)

These disclaimers MUST be present in the listed surfaces:

| Surface | Required Disclaimer |
|---|---|
| Onboarding (first screen) | "Bendro provides general flexibility and mobility content. It is not medical advice. Consult a qualified healthcare provider before starting any new exercise program." |
| Routine start screen | "Listen to your body. Stop if you feel sharp pain." |
| Pain feedback prompt (post-session) | "If you are experiencing significant pain, please consult a healthcare provider." |
| AI-generated routine cards | "Generated by AI based on your goals. Not a substitute for professional guidance." |
| Pre-existing-condition path | "Stretches involving [body part] may be unsafe with certain conditions. Skip or modify if any cause discomfort." |
| Marketing pages mentioning "pain relief" | "Results vary. Bendro does not treat or cure any medical condition." |

The disclosure copy lives in `src/lib/disclaimers.ts` (single source of truth, to be added in Phase 11).
Every surface that needs a disclaimer imports from this module — never inline.

## Pain Feedback Flow (Hard Requirement)

Post-session pain self-rating uses a 0–10 scale.

| Rating | Required Behavior |
|---|---|
| 0–3 | Normal flow — store rating, continue |
| 4–6 | Show "ease back next time" hint, suggest gentler alternative |
| 7–10 | Stop further auto-recommendation of similar stretches; show medical-guidance disclaimer; flag the routine in personalization to deprioritize |

The thresholds are constants in `src/services/safety.ts` (to be added in Phase 11). Never inline.

## Pre-Existing Condition Gating

Onboarding asks (yes/no, not free-text):
- Recent injury (≤ 6 weeks)
- Surgery in the last 6 months
- Diagnosed condition affecting joint/spine mobility
- Pregnancy (current or last 6 weeks)

A "yes" on any item:
1. Surfaces a "speak with a healthcare provider before continuing" interstitial.
2. Stores a `safety_flag` on the user record (do NOT store the underlying reason — privacy).
3. Filters the routine catalog to exclude routines tagged `caution: deep-spine`, `caution: high-load`, `caution: prone`.

Filter implementation lives in `src/services/personalization.ts`. The flag is set once and cleared only by user action.

## Copyright Rules (Stretch Descriptions and AI Prompts)

1. Do not copy verbatim stretch descriptions from textbooks, web sources, or competitor apps. Paraphrase in our own voice.
2. Do not use copyrighted illustrations or photo references without license. SVG/3D content must be either original or under permissive license.
3. AI-generated routines MUST NOT name specific copyrighted protocols (e.g., proprietary trademarked methods) unless the trademark holder has licensed it.
4. Style and general technique descriptions are not copyrightable — describing how a hamstring stretch works is fine.

## Defamation / Real-Person Rules

1. Do not generate AI content that names a specific real PT, doctor, or athlete as endorsing a routine.
2. Do not generate fake testimonials.
3. Marketing testimonials must be from real, consented users — store the consent record alongside the quote.

## Platform / Health-Data Compliance

When integrating health platforms (Apple HealthKit, Google Fit, Garmin, etc.) — current state: NOT YET INTEGRATED:

1. Read scopes ONLY (not write) until ADR justifies write access.
2. Display the platform's required disclosure verbatim before requesting access.
3. NEVER store raw health-platform data on our server without explicit user opt-in for "cloud sync."
4. HKQuantityTypeIdentifierBodyMass, sleep data, and any clinical data category require explicit per-category consent UI.
5. If we ever store regulated health data, we trip into HIPAA scope (US) and GDPR special-category data (EU). Block this until ADR + legal sign-off.

## Camera & Pose Privacy (Coupled to SECURITY_RULES.md)

Pose detection runs entirely client-side. Frames and landmark data NEVER leave the device.

Re-stated here because it is ALSO a health-safety concern — recording someone's body without their explicit knowledge is both a privacy and a safety risk:

1. Camera access requires an explicit user gesture (button click). Auto-prompting is forbidden.
2. The camera preview MUST be visible to the user whenever it is active (no hidden capture).
3. There is no recording feature in v1. If we add one (Phase 14+), it stores client-side only and requires a separate consent flow.
4. If we ever add server-side pose analysis (e.g., form-feedback ML model), document in ADR FIRST and add a dedicated consent step.

## Emergency Language

The product must NEVER include language that delays a user from seeking emergency care.

If our pain-feedback flow detects a likely emergency (severe sudden pain, numbness, loss of sensation — not currently inferred but planned for Phase 11), the screen MUST surface "If this is a medical emergency, call your local emergency services" prominently and unblockably.

## Enforcement

Enforced by:
1. `src/lib/disclaimers.ts` — single source of truth for disclaimer copy (Phase 11)
2. `src/services/safety.ts` — pain thresholds + gating logic (Phase 11)
3. `src/services/personalization.ts` — safety_flag-based filtering (Phase 6 / Phase 11)
4. `pr-reviewer` agent — reviews any user-facing copy or AI-generated content for compliance with this file
5. `security-check` skill — checklist includes health-disclaimer presence
6. Audit trail in the future `safety_events` table — pain feedback ≥ 7 always logged (Phase 11)
