# Security Rules — Bendro

> Zero-tolerance security rules. Violations block PRs and trigger security review.
> These apply to every route, every service, every commit.

---

## Authentication Rules

1. NextAuth.js session required on every API route that reads or writes user-owned data.
2. Session cookie is `httpOnly`, `secure` (production), `sameSite=lax`.
3. Public routes allowed without session: `/api/stretches` (catalog read), `/api/routines` (catalog read, non-owned).
4. `userId` comes ONLY from the server-side NextAuth session. NEVER from request body or query.
5. Session secrets rotate on NEXTAUTH_SECRET change — configure via env only.

## Authorization Rules

1. Route handlers call `requireSession()` helper (to be added in `src/lib/auth.ts`).
2. Resource ownership checks happen in the service layer — route handlers never compare `userId`.
3. Cross-user access returns 404 (not 403) to prevent user enumeration.
4. Premium-gated resources check `user.subscriptionStatus` at the service layer.

## Data Rules

1. Every query on user-scoped tables (`sessions`, `favorites`, `streaks`, user profile) MUST filter by `userId` from session.
2. All DB access through Drizzle with parameterized queries — Drizzle's query builder is safe by default.
3. NEVER use `sql.raw()` or string concatenation with user input. If `sql.raw` is truly required, inputs must be allowlist-validated.
4. User input validated with Zod at every route boundary before reaching the service layer.
5. DTO mapping in the service layer — never return raw Drizzle rows to the client (strips internal columns).

## Secrets Rules

1. Zero secrets in code, `.env*` files (except `.env.example`), or git history.
2. `.env.example` contains only placeholder values. `.env.local` is gitignored.
3. Production secrets live in Vercel Environment Variables, scoped to Production (never Preview/Development unless explicitly needed).
4. `detect-secrets scan` runs in CI on every PR.

## Logging Rules

1. NEVER log: passwords, email addresses, session cookies, API keys, Stripe customer/subscription IDs, OAuth tokens.
2. Safe to log: `userId` (UUID), `routineId`, `sessionId`, request path, status code, timing.
3. Log at appropriate levels: ERROR (unhandled exceptions), WARN (auth failures, 4xx), INFO (business events), DEBUG (internal flow).
4. Production logs go to Vercel + Sentry — never `console.log` user input in production code.

## Stripe Rules

1. Webhook endpoint (`/api/stripe/webhook`) MUST verify signature with `STRIPE_WEBHOOK_SECRET` on every request.
2. Webhook handler is idempotent: repeated delivery with the same `event.id` produces no duplicate side effect.
3. Never trust Stripe customer/subscription IDs from client requests — always look up via session user's `stripeCustomerId`.
4. Stripe Checkout session creation: always set `customer` or `customer_email` from session, never from client input.
5. Subscription status updates come ONLY from webhooks, never from client-side calls.

## Camera / Media Rules

1. Camera access is opt-in, gated by an explicit user action (button click in player).
2. Pose landmarks and camera frames NEVER leave the client — no upload, no analytics beacons carrying pose data.
3. If a future feature requires server-side pose analysis, document the data flow in an ADR before implementation.
4. Player camera page announces the privacy model in visible UI copy.

## Dependency Rules

1. `pnpm audit --audit-level=high` runs in CI. High/Critical findings block merge.
2. Container scans (if containerized) use Trivy.
3. No packages with GPL-3.0 or AGPL licenses without review.

## CORS and Transport Rules

1. CORS: Next.js default (same-origin). Any cross-origin need is an explicit allowlist entry — never wildcard.
2. All external HTTP calls have explicit timeouts (default 10s, max 30s). Use `AbortSignal.timeout()`.
3. SSRF protection: if the app fetches user-provided URLs (e.g., custom avatar), validate against an allowlist.
4. File uploads (future): max 5MB for avatars, MIME validation, stored in S3/Vercel Blob — never the DB.

## Content Safety Rules

1. AI-generated routines (when added) MUST pass a safety check before returning to the user — no exercises that are dangerous without medical context.
2. All AI-generated routines carry the disclaimer: "Generated by AI. Listen to your body. Stop if you feel pain. Not medical advice."
3. User-reported pain ≥ 7 suppresses AI suggestions for that body area until the user updates.

## Rate Limiting Rules

1. Public catalog endpoints: 60 req/min per IP.
2. Authenticated mutations (create session, update streak): 30 req/min per `userId`.
3. Rate limit storage: Vercel KV or Upstash Redis (to be added in Phase 12). For dev, in-memory counter is acceptable.
